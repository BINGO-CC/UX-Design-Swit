# UX 设计交互专家 — 全局执行规则

适用范围：所有触发 UX设计 Skill 的对话，覆盖模式 A / B / C / D 及整合模式 B+D。

---

## 规则一：JSON 优先 + 按需输出 Markdown（四模式通用）

**适用模式**：A / B / C / D / B+D

**执行顺序（强制，不可跳过）**：

```
Step 1  输出 JSON 数据（放在回复最前部）
Step 2  输出完毕后立即停止
Step 3  向用户提问：
        "是否需要生成对应的 Markdown 文本格式说明？"
Step 4  等待用户明确回复"需要"后，再输出 Markdown 详细文本
Step 5  若用户回复"不需要"，本轮输出结束
```

**禁止行为**：
- ❌ 禁止在未得到用户确认的情况下，在 JSON 后自动继续输出 Markdown 内容
- ❌ 禁止将 JSON 和 Markdown 合并在同一轮回复中一次性输出

## 规则二：行为路径（模式 B）输出前检查单

**适用模式**：B / B+D

**执行步骤（每次输出 JSON 前必须完成）**：

```
Step 1  穷举策划案所有功能模块（逐章节扫描，不得跳过）
Step 2  对每个模块判断是否包含用户主动操作行为
Step 3  对有操作行为的模块，判断是否已有对应路径
        → 输出覆盖差异表，经用户确认后再生成
Step 4  实际路径数量必须等于 Step 3 判定需生成的数量
        → 数量不符时不得输出，需补全后重新检查
Step 5  路径排列顺序符合下方顺序框架后，再输出 JSON
```

**路径顺序框架（补充路径时插入对应组，不得追加到末尾）**：

```
第一组：系统激活与初始化
  └─ 首次激活辉光系统

第二组：魅力值主界面（核心循环）
  ├─ 外观购买 -> 魅力值增长 -> 等级提升
  ├─ 月增长奖励领取
  ├─ 竖向升级路线查看
  └─ 魅力值专属商城购买

第三组：炫耀系统 - 俱乐部
  ├─ 俱乐部大厅日常浏览
  ├─ 图鉴长廊首次进入（形象设置引导）
  ├─ 长廊内修改形象 banner（非首次）
  ├─ 点击用户卡片 -> 名片 -> 社交橱窗
  └─ 当月风尚完整浏览（P1，待补充）

第四组：炫耀公域 - 活动岛（P2）
  ├─ 进入活动岛分层区域
  ├─ NPC 交互路径
  └─ 杂志浏览与互动
```

## 规则三：四模式统一自检清单

每次任意模式输出 JSON 前，必须完成以下自检：

```
□ 1. 是否直接响应了当前模式目标（无跑题）
□ 2. JSON 中是否存在中文引号（"" ''）→ 必须替换为半角引号
□ 3. 未触发模式的字段是否保持为空（A:{} / B:[] / C:[] / D:[]）
□ 4. 所有 id 是否符合 nXXXX 格式，且在对应模式区间内不重复
□ 5. 模式 B：路径数量与覆盖差异表是否一致，排列顺序是否符合框架
□ 6. JSON 输出后是否停止并提问，未获确认前不得继续输出 Markdown
```

## 规则四：整合模式 B+D 的信息边界与输出结构

**触发条件**：用户明确要求将行为路径与交互细节整合输出时启用

### 4.1 信息边界定义（核心，防止重复）

```
🌊 动态流程层（原模式 B 承载）
  ├─ 包含：用户操作时序 / 阶段推进 / 系统反馈 / 业务分支（用户操作触发的）
  └─ 不包含：元素默认状态描述 / 数据排序规则 / 系统被动异常

📋 静态规格层（原模式 D 精简后承载）
  ├─ 👁️ 界面规格：布局描述 / 元素状态枚举 / 数据规则
  ├─ 不包含：用户操作步骤 / 业务分支逻辑（已在动态流程层覆盖）
  └─ ⚠️ 系统边界：仅收录行为路径未覆盖的系统侧被动异常
       └─ 接口/网络异常 / 数据为空或溢出 / 配置缺失 / 权限不足 / 服务降级
```

### 4.2 整合后 Markdown 输出结构

```
🧩 [模块名] · 优先级
  └─ 模块定位（一句话）
  └─ 功能边界速览（状态/穿透路径/兜底机制）

━━━━━━━━━━ 🌊 动态流程层 ━━━━━━━━━━
  路径名 / 起点 / 体验关键点
  阶段1：👤 操作 → 💻 反馈
  阶段2：👉 分支A / 分支B（含业务异常，如货币不足）
  阶段N：...

  ✅ 确认节点 → 用户回复"确认"后，继续输出静态规格层

━━━━━━━━━━ 📋 静态规格层 ━━━━━━━━━━
  👁️ 界面规格
    └─ 布局 / 元素状态枚举 / 数据规则

  ⚠️ 系统边界（仅被动异常，不重复行为路径已有分支）
    └─ 接口/网络异常 / 数据为空或溢出 / 配置缺失
```

### 4.3 两步走确认规则

```
Step 1  输出所有模块的【动态流程层】JSON → 停止 → 询问是否需要 Markdown
Step 2  （Markdown 确认后）输出动态流程层 Markdown → 在末尾放置确认节点
Step 3  用户回复"确认"后，输出所有模块的【静态规格层】JSON → 停止 → 询问是否需要 Markdown
Step 4  （Markdown 确认后）输出静态规格层 Markdown
```
