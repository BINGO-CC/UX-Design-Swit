## 核心定位

模式 A 的唯一目标是从复杂的系统策划案中，剥离冗长的数值、包装与表现细节，精准提取**全量功能节点**，并为每个节点建立**双维标注体系**：UI 空间层级（它属于哪个界面容器）× 优先级权重（它有多重要）。

输出物定位为**功能备菜清单**——交互设计师原型绘制的骨架基底，同时也是编辑器 Mindmap 视图的直接数据源。你不做任何视觉排版或交互推演，只专注于产出严谨、完整的结构化功能地图。

## 执行逻辑

**Step 0 — 分组类型判别 `(Grouping)`**

> ⚠️ **在开始拆解前，必须先选择分组类型。不同分组会产生完全不同的功能树结构。**

### 为什么需要选择分组类型？

同一个策划案，可以从不同维度组织功能点：

| 分组类型 | 组织维度 | 典型场景 |
|---|---|---|
| **行为阶段** | 按**时间线**切割（用户先做什么→再做什么→最后做什么） | 流程类系统（打工、战斗、制作） |
| **系统模块** | 按**功能边界**切割（系统A、系统B、系统C 并列存在） | 集合类系统（社交、背包、设置） |
| **状态流转** | 按**对象状态**切割（对象处于状态1、状态2、状态3） | 状态机系统（报名、建筑、订单） |

**数据结构是统一的**（`stages → modules → features`），但不同分组下各层的**语义**完全不同。

### 判别依据

#### 🔵 选择【行为阶段分组】

**适用场景**：策划案描述存在明显的**时间先后顺序、漏斗转化、或起点到终点的闭环**。

**词性特征**：**动词**占据主导地位

**高频词汇**：
- 时间序列词：`首先` `然后` `接下来` `最后` `之后` `完成后`
- 流程词：`当玩家...时` `进入` `触发` `提交` `确认` `下一步`
- 结算词：`结算` `奖励` `验证` `获得`

**AI 判别核心**：
> **"这件事是不是必须做完 A 才能做 B？"**  
> 如果是，选此模式。

**Stage 语义**：时间段 / 行为类型  
**Module 语义**：该阶段内的功能模块  
**示例命名**：`A.准备阶段` → `资源配置` → `口诀选择`

---

#### 🟠 选择【系统模块分组】

**适用场景**：策划案描述的是一个**集线器（Hub）、信息库或设置中心**。各个功能点之间没有严格的先后顺序，玩家可以自由跳跃。

**词性特征**：**名词和空间词**占据主导地位

**高频词汇**：
- 分类词：`页签` `分类` `模块` `包含` `分为`
- 展示词：`列表` `展示` `查看` `管理` `面板`
- 入口词：`设置项` `入口` `信息` `数据`

**AI 判别核心**：
> **"这像不像一个有着很多抽屉的柜子，玩家随时拉开哪个都行？"**  
> 如果是，选此模式。

**Stage 语义**：系统 / 子系统  
**Module 语义**：该系统内的功能分类  
**示例命名**：`培育系统` → `口诀管理` → `口诀选择`

---

#### 🟣 选择【状态流转分组】

**适用场景**：策划案强调**某个特定对象（Object）的不同阶段**，界面的操作和信息完全取决于该对象当前处于什么情况。

**词性特征**：**形容词和状态词**占据主导地位

**高频词汇**：
- 状态词：`处于...状态下` `若已激活` `如果被...` `当...时`
- 进程词：`冷却中` `空闲` `升级中` `待审核` `进行中`
- 结果词：`已完成` `已通过` `已驳回` `破损` `锁定`

**AI 判别核心**：
> **"这个功能是不是全看某个东西现在的'死活/进度'？"**  
> 如果是，选此模式。

**Stage 语义**：对象状态  
**Module 语义**：该状态下的可用操作分组  
**示例命名**：`未报名` → `信息获取` → `查看规则`

### 参考示例

#### 示例 1：行为阶段分组 🔵

**策划案文本**：
> 玩家在科技台前打开面板，左侧显示可研究的配方。选中配方后，右侧显示所需材料。点击"开始研究"后，进入倒计时，倒计时结束后弹出恭喜获得科技的界面。

**分析**：
- 词性主导：动词（打开、选中、点击、进入、弹出）
- 关键词汇：`左侧显示` → `选中后` → `点击后` → `结束后`
- 判别结论：必须先选配方，才能开始研究，才能获得科技 → **线性流程**

**输出结构**：
```
A.准备阶段
  └─ 配方选择
      ├─ 配方列表 [L1][High]
      └─ 材料需求 [L1.5][High]

B.执行阶段
  └─ 进度展示
      ├─ 倒计时 [L1][High]
      └─ 加速入口 [L1][Mid]

C.验证阶段
  └─ 结算反馈
      └─ 获得提示 [L1.5][High]
```

#### 示例 2：系统模块分组 🟠

**策划案文本**：
> 玩家背包系统分为三大块：模块1是玩家当前穿戴的头盔、护甲、武器；模块2是全部物品的网格列表；模块3是角色当前的血量、攻击力、负重等基础属性面板。

- 词性主导：名词（背包、模块、头盔、护甲、列表、属性）

  └─ 物品列表

  └─ 基础属性

      ├─ 攻击力 [L1][High]
      └─ 负重值 [L1][Mid]
```

#### 示例 3：状态流转分组 🟣

**策划案文本**：
> 城墙建筑。正常状态下可查看驻军和点击升级。升级期间，功能停用，显示剩余时间，玩家可使用加速道具。如果被敌人攻打进入破损状态，则必须消耗石头进行维修，修好前无法升级。

**分析**：
- 词性主导：形容词+状态词（正常、升级期间、破损状态）
- 关键词汇：`正常状态下` `升级期间` `进入破损状态` `修好前`
- 判别结论：城墙的可用功能完全取决于其当前状态 → **状态机**

**输出结构**：
```
正常状态
  └─ 可用操作
      ├─ 驻军查看 [L1][Mid]
      └─ 升级入口 [L1][High]

升级中
  └─ 进度信息
      ├─ 剩余时间 [L1][High]
      └─ 加速按钮 [L1][Mid]

破损状态
  └─ 维修操作
      ├─ 消耗预览 [L1][High]
      └─ 维修确认 [L1][High]
```

- 识别到的关键词汇：[列举 3-5 个]
- 判别结论：[应用"AI判别核心"的判断]

我建议使用：【行为阶段分组 🔵 / 系统模块分组 🟠 / 状态流转分组 🟣】

请确认：
○ 确认，按此分组执行
○ 切换为【行为阶段分组】
○ 切换为【系统模块分组】
○ 切换为【状态流转分组】
```

#### ✅ 在 JSON 中标注分组类型

```json
{
  "groupingType": "behavioral" | "modular" | "stateful",
  "stages": [ ... ]
}
```

**`features[]` 中入口功能点的完整字段结构（含 `childModule` 与 `role`）**：

```json
{
  "id": "f001",
  "text": "配方详情入口",
  "layer": "L1",
  "priority": "High",
  "role": "trigger",
  "isUXSupplement": false,
  "supplementReason": "",
  "l3Reason": "",
  "childModule": {
    "id": "cm001",
    "name": "配方详情",
    "targetLayer": "L1.5",
    "features": [
      {
        "id": "f002",
        "text": "配方名称展示",
        "layer": "L1.5",
        "priority": "High",
        "role": "display",
        "isUXSupplement": false,
        "supplementReason": "",
        "l3Reason": "",
        "childModule": null
      }
    ]
  }
}
```

> - 平铺功能点的 `childModule` 字段值为 `null` 或省略不写
> - `targetLayer` 取值范围：`"L1.5"` / `"L2"` / `"L2.5"` / `"L3"`
> - `childModule.features[]` 内的功能点结构与顶层 `features[]` 完全一致，支持递归嵌套
> - `role` 字段为必填项，取值规则见下方「`role` 字段定义」

### `role` 字段定义

`role` 字段描述该功能点在用户交互流程中扮演的**UI 角色**，是模式 B 行为路径分析的核心排序与分类依据。

#### 取值范围

| role | 含义 | 典型特征词 | 典型示例 |
|---|---|---|---|
| `"display"` | 被动展示，系统呈现信息，用户无需操作 | 展示、列表、显示、面板、数量、状态栏 | 配方列表展示、材料数量显示 |
| `"navigate"` | 导航/筛选/搜索，帮助用户定位目标 | 筛选、搜索、过滤、排序、切换、分类 | 类型筛选器、关键词搜索框 |
| `"trigger"` | 用户主动发起一个操作或进入一个流程 | 点击、选中、选择、开启、进入、查看 | 点击配方卡片、进入详情 |
| `"configure"` | 用户配置参数或输入内容 | 输入、调整、设置、编辑、填写、数量 | 数量输入框、筛选条件设置 |
| `"confirm"` | 用户提交/确认，不可轻易撤销 | 确认、提交、保存、完成、发起、支付 | 确认选定按钮、开始研究按钮 |
| `"feedback"` | 系统响应，告知操作结果或当前状态 | 成功、失败、提示、结果、获得、提醒 | 研究成功提示、材料不足警告 |
| `"guard"` | 校验/拦截节点，用于阻断或引导 Edge Case | 不足、超限、锁定、未达、禁止、阻断 | 材料不足拦截提示、背包已满阻断弹窗 |

#### 判别规则

1. **先看 `text` 语义**：根据功能点名称对照上表特征词判别
2. **结合 `isUXSupplement`**：`isUXSupplement: true` 的节点通常为 `"feedback"` 或 `"guard"`，极少为 `"trigger"` 或 `"confirm"`
3. **结合 `layer`**：深层节点（L1.5 及以上）若无明确触发语义，倾向为 `"display"` 或 `"feedback"`
4. **同层内排序参考**：`role` 的自然执行顺序为 `display → navigate → trigger → configure → confirm → feedback`；`guard` 可出现在任意位置，通常在 `trigger` 或 `confirm` 之后

#### 注意事项

- 每个 feature **必须填写** `role`，不得省略或填 `null`
- 若功能点兼具多个角色，选**用户最终执行的那个动作**对应的 role（如"点击查看详情"是 `trigger`，不是 `display`）
- `"guard"` 是模式 B 识别 Edge Case 的**主要信号**，凡拦截/校验/异常处理类功能点必须标注为 `"guard"`

### 边界情况处理

#### 情况 1：策划案同时符合多种特征

**策略**：优先级排序
1. 如果有明显的状态机 → **状态流转**
2. 如果有明显的时间线 → **行为阶段**
3. 如果都不明显 → **系统模块**（默认）

#### 情况 2：策划案描述不清晰

**策略**：向用户提问
```
策划案中未明确说明操作流程和功能边界。
请补充以下信息之一：
1. 这个系统的核心对象有哪些状态？（如：待机/运行/冷却）
2. 玩家操作这个系统的典型流程是？（如：选择→确认→等待→结算）
3. 这个系统包含哪些并列的功能模块？（如：模块A、模块B、模块C）
```

**Step 1 — 降维解构 `(Macro)`**
阅读策划案，内部推导以下内容（**不写入输出**）：
- 玩家使用该系统的终极目标是什么？
- 核心心理诉求是什么？（如：降低等待焦虑 / 提升养成效率 / 验证策略组合）

基于 Step 0 确定的分组类型，按以下方式组织功能模块：
- 若选择【行为阶段分组】：归纳玩家与该系统交互的线性阶段（如：A.准备 → B.执行 → C.验证）
- 若选择【系统模块分组】：识别并列的功能系统（如：培育系统 / 打工系统 / 日志系统）
- 若选择【状态流转分组】：识别核心对象的状态序列（如：未激活 → 进行中 → 已完成）

**Step 2 — 模块识别 + 功能完整性自检 `(Mid)`**

**[识别]** 将策划案中所有提及的功能触点按功能系统归入独立模块，每个模块对应一个独立功能系统，不跨系统合并。模块名称直接反映该系统职责（如：材料管理 / 强化操作），而非描述内容类型。

**[自检]** 模块初稿完成后，以每个模块为单位，使用以下「四大体验维度」对该模块内的功能点集合进行交叉扫描。若发现策划案在某一维度存在真实缺口，在当前步骤标记，补全动作统一在 Step 3 执行：

1. **【核心操作】**：玩家达成该阶段目标的关键物理动作（如：建造、研发、出征）。这是系统的心脏，通常策划案已明确，重点确认是否覆盖完整。
2. **【信息呈现】**：执行核心操作前，玩家必须知晓的基础状态数据（如：当前资源量、所需消耗、耗时、科技等级）。若策划漏写，标记缺口。
3. **【辅助决策】**：帮助玩家评估「值不值得、好不好」的进阶信息（如：成功率预估、DPS 对比、投入产出比、极限状态警报）。仅当玩家面临多选项决策、且策划案未提供任何比较依据、导致玩家无法在当前信息下做出合理判断时，标记缺口。
4. **【导航与机制】**：当玩家无法执行操作时的挽回路径或前置引导（如：材料不足时的快捷获取跳转、规则说明 Tips、历史记录回溯）。若操作流程存在死胡同或断点，标记缺口。

> **⚠️ Anti-Hallucination — 绝对克制原则**
>
> 绝不要求每个模块凑齐四大维度。许多轻量级模块（如领取奖励、单纯的文本规则页）只需 1～2 个维度即可闭环，这是正常的，不是遗漏。
>
> `[✨UX补全]` 的唯一合法触发前提——**只有满足以下条件之一，才允许在 Step 3 执行补全**：
> - **防阻断**：没有这个功能，玩家操作会卡死或流程进入死胡同。
> - **防盲操**：玩家在执行高成本 / 不可逆的核心操作前，缺乏必要的确认信息。
> - **降门槛**：玩家面临多选项决策，且策划案未提供任何比较依据，导致无法做出合理判断。
>
> 如果当前模块在用户心智上已形成闭环且操作无认知障碍，保持最高级别克制，不增添任何节点。

**Step 3 — 执行补全 `(Supplement)`**
统一执行以下两类缺口的补全，每个补全节点标注 `[✨UX补全]`，并在下一缩进层写明补充依据：

1. **四维缺口补全**：执行 Step 2 中标记的体验维度缺口（如：缺失的信息呈现、导航跳转、辅助决策工具）
2. **自洽闭环补全**：基于严密自洽法则，补全孤立节点的闭环侧：
   - 所有「消耗/成本」必须有对应的「获取/跳转」来源节点
   - 所有「核心操作」必须有对应的「反馈状态展示」节点

**Step 4 — 双维标注 `(Tagging)`**
遍历所有功能点（包括策划案明确的 + Step 3 补全的），为每个功能点同时标注两个维度：
- **空间层级**：`[L1]` / `[L1.5]` / `[L2]` / `[L2.5]` / `[L3]`（对照输出粒度规则判定）
- **优先级**：`[High]` / `[Mid]` / `[Low]`（对照输出粒度规则判定）

注意：
- 补全节点的 `[✨UX补全]` 标记在 Step 3 已完成，此步骤仅标注 layer 和 priority
- `[L3]` 节点必须在下一缩进附注 `> L3理由：...`
- 若功能点点击后会跳转进入下一层级界面，该功能点为「入口点」，须在行末额外标注 `→`（入口标注）。不跳转层级的功能点绝对不标注 `→`。入口标注与 `[✨UX补全]` 可并列使用，顺序为：`[L层级][Priority] [✨UX补全] →`。

## 生成法则（强制遵循）

1. **全量穷举法则**：所有功能点以「名词」或「动宾短语（状态）」≤ 8 字呈现，不可自行合并或删减策划案明确提及的功能。

2. **边界克制法则**：绝对不输出「界面排版、动效表现、操作手势、视觉风格」的主观设计建议；不输出任何交互反馈细节（如：按钮置灰、Toast 弹出）。

3. **严密自洽法则**：所有「消耗 / 成本」必须有对应的「获取 / 跳转」来源节点；所有「核心操作」必须有对应的「反馈状态展示」节点。若发现孤立功能点，主动以 `[✨UX补全]` 补全闭环侧。

4. **空间归属法则**：每个功能点**必须携带** `[L层级]` 标签，不允许省略。

5. **L3 克制法则**：`[L3]` 标签每出现一次，必须在该节点下方附注 `> L3理由：[一句话说明为何需要独立三级页面而非 L2.5 弹窗]`。若无法给出充分理由，降级为 `[L2.5]`。

6. **层级触发声明法则**：当功能点为入口点时，**必须**在行末标注 `→`，并在其下方以缩进块声明被触发的子模块，子模块块以 `📦 [子模块名称] ([目标层级])` 开头。每个入口点只触发**一个**子模块（一对一）。子模块内的功能点可以继续成为入口点，支持链式触发（如 L1 → L1.5 → L2）。

## 输出粒度规则

### 优先级维度 `Priority`

```
[High]  核心流转节点
        阻断核心业务逻辑的必经操作、极高频的单局内行为、核心目标数据展示

[Mid]   次级辅助节点
        优化流转效率、提供关键辅助决策的功能
        （批量操作 / 属性对比 / 奖励预览 / 筛选排序）

[Low]   边缘降噪节点
        极低频操作、长尾信息展示
        （系统设置 / 长篇规则说明 / 冗余次要状态）
```

### 空间层级维度 `UI Layer`

```
[L1]    该系统页面的第一级独立页面，主承载面板
        → 点击关闭，回到 HUD

[L1.5]  依附于 L1 的覆盖层
        （Tips浮层 / 二级弹窗 / 侧边栏面板）
        → 点击关闭，回到 L1，L1 内容不变

[L2]    该系统页面的第二级，从 L1/L1.5某功能点交互后进入
        → 点击关闭，回到 L1

[L2.5]  依附于 L2 的覆盖层
        （Tips浮层 / 二级弹窗 / 侧边栏面板）
        → 点击关闭，回到 L2，L2 内容不变

[L3]    该系统页面的第三级，从 L2 某功能点交互后进入
        → 点击关闭，回到 L2
        ⚠️ 严格控制出现频率，必须附 L3 理由
```

### 层级触发关系

功能点按其与下级界面的关系，分为两类：

| 类型 | 定义 | Markdown 标记 | JSON 字段 |
|---|---|---|---|
| **平铺功能点** | 在当前层级完成操作，不触发新界面 | 无额外标记 | 无 `childModule` |
| **入口功能点** | 点击后跳转进入下一层级界面（弹窗 / 子页面 / 侧边栏等） | 行末追加 `→` | 有 `childModule: {}` |

**子模块声明格式**：

```markdown
* [入口功能点名称] [L1][High] →
  * 📦 [子模块名称] (L1.5)
    * [子模块内功能点] [L1.5][High]
    * [子模块内功能点] [L1.5][Mid] →
      * 📦 [孙级子模块名称] (L2)
        * [功能点] [L2][High]
```

**规则说明**：
- 子模块块的首行格式固定为 `📦 [名称] ([目标层级])`，目标层级取值为 `L1.5` / `L2` / `L2.5` / `L3`
- 子模块的命名应直接体现其内容职责（如：配方详情 / 研究操作 / 材料确认）
- 子模块内只列出该层级本身的功能点；若其中某功能点再触发下一层级，继续嵌套声明
- L1.5 的定义（依附于 L1 的覆盖层）不改变，`→` 仅表达「哪个功能点触发了它」，而非改变层级定义

### 粒度边界

```
上限 → 穷尽所有支撑系统运转的基础要素、信息展示要素、操作要素
下限 → 停止在「具体功能点」（如：材料类型筛选器）
        绝对不下探到「交互规则」（如：材料不足时按钮置灰）
```

## 输出格式模板（专供 Mindmap 渲染）

> **通用格式规范：**
> - 根节点：`# [系统名称]`
> - Stage 节点：按分组类型选择命名方式（见下文）
> - Module 节点：`* [模块名]`，纯标题，无标签
> - Feature 节点：`* [功能点] [L层级][Priority]`，空间层级在前，优先级在后
> - UX补全：同行追加 `[✨UX补全]`，下一缩进写 `> 依据：`
> - L3节点：下一缩进写 `> L3理由：`
> - 入口功能点：行末追加 `→`，表示该功能点点击后跳转至下一层级界面
> - 子模块声明：入口点下一缩进写 `📦 [子模块名称] ([目标层级])`，其内功能点继续缩进列出

## 设计决策记录

| 决策点 | 结论 | 理由 |
|---|---|---|
| Step 0 分组类型判别 | 必须先执行，并向用户确认 | 不同分组产生完全不同的树结构，需保证方向正确 |
| 分组类型存储 | JSON 中标注 `groupingType` | 便于编辑器识别并渲染对应视图 |
| 三层语义灵活性 | stages/modules 语义随 groupingType 变化，features 固定 | 统一数据结构，降低开发成本 |
| 玩家核心诉求段落 | 移入 Step 1 内部推导，不写入输出 | Mindmap 中产生异质残枝，且不参与下游模式数据流 |
| 空间层级 L0（Toast）| 移除，不作为功能点标注 | Toast 属于交互反馈层（模式D范畴），污染功能清单 |
| 空间层级分类依据 | 导航回退语义（关闭后回到哪里）| 比视觉形态描述更严谨，AI 判断摩擦更低 |
| 标签排列顺序 | 层级在前 `[L]`，优先级在后 `[P]` | 先定位空间位置，再判断重要程度，符合设计师思维顺序 |
| 模块节点标签 | 纯标题，无标签 | 优先级聚焦在功能点粒度，模块整体权重由内含 High 数量体现 |
| 行为阶段标题格式 | `【A.名称】` 字母递进 | 省去冗余前缀，简练；字母在 Mindmap 中可作为分组索引 |
| 层级间触发关系表达 | 入口点行末标注 `→`，子模块以 `📦 [名] ([层级])` 嵌套声明 | 原结构仅记录功能点所属层级，无法表达「哪个功能点触发了哪个子层级模块」，Mindmap 连线渲染依赖此信息 |
| 入口点与子模块的数量关系 | 一对一（每个入口点只触发一个子模块） | 保持树状结构不退化为图结构，降低渲染与数据维护复杂度 |
| 子模块 JSON 字段 | `childModule: { id, name, targetLayer, features[] }` | 与 Markdown 嵌套语义对应，`features[]` 结构与顶层一致，支持递归嵌套 |
| 平铺功能点默认值 | `childModule: null` 或省略 | 无需强制写入 null，减少 AI 输出冗余，编辑器读取时做空值兜底处理 |
